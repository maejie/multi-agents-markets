# multi-agents-markets

自動車の製造から廃棄までの VLT (Vehicle Lifetime Value) を学習目的で可視化する、ブラウザ実行型のマルチエージェントシミュレーター。

このプロジェクトは、自動車販売向けの金融事業に関わる人が、中古車価格がどのように形成されるか、何が変動要因になりうるかを、経営シミュレーションゲーム風の UI で学べることを目的とする。

## 1. プロジェクトの目的

このシミュレーターで学習したい対象は以下。

- 中古車価格の形成プロセス
- 車齢、需給、買い替え行動が価格に与える影響
- 新車販売、中古流通、廃棄までを通した車両ライフサイクル
- VLT の観点から見た市場全体の流れ

MVP では、説明可能性と再現性を優先し、価格決定やエージェント行動を単純化したルールベースのシミュレーションとして実装する。

## 2. 技術スタック

| 役割 | 採用技術 |
|------|----------|
| ビルドツール | Vite |
| UI | Svelte 5 |
| 言語 | TypeScript |
| シミュレーション実行 | Web Worker |
| Worker 通信 | Comlink |
| 2D 描画 | PixiJS |
| グラフ | Chart.js |
| 乱数 | seedrandom |

アーキテクチャ概要:

```text
Svelte UI (main thread)
  ├─ PixiJS: 市場ビュー描画
  ├─ Chart.js: 時系列グラフ
  └─ Comlink
	  └─ Web Worker
		  └─ TypeScript simulation engine
			  └─ seedrandom による再現性確保
```

## 3. ドメインモデル

### 3.1 時間単位

- 1 ステップ = 1 ヶ月
- 人も車両も年齢は「月」で統一する

### 3.2 エージェント

#### 個人顧客

状態:

- id
- 年齢 (月)
- 資金
- 保有車両 id
- 保有開始月
- 利用予定期間 (36-84 ヶ月)
- 売り注文中フラグ

ルール:

- 216 ヶ月超かつ未保有の個人は、毎月 5% の確率で車が必要になる
- 車が必要になった場合、50% の確率で新車、50% の確率で中古車を選ぶ
- 中古在庫がない場合は新車購入にフォールバックする
- 購入後 36 ヶ月以降は毎月 3% の確率で売却意思を持つ
- 保有期間が 84 ヶ月に達した場合は強制的に売却意思を持つ
- 売却が成立したら、同じ月に次の車両を購入する
- 840 ヶ月に達すると市場から退出する
- 3 ヶ月に 1 回、新しい個人が 1 人誕生する

#### メーカー

- 新車注文が入った時のみ生産する
- 新車価格で即時販売する
- MVP では生産能力制約なし

#### リサイクラー

- 寿命到達車両を処分する
- 所有者退出時に残存する車両も処分する
- MVP では廃棄報酬なし

### 3.3 車両

状態:

- id
- 車齢 (月)
- 寿命 (180 ヶ月)
- 新車価格 (180)
- ステータス: `in_use` / `for_sale` / `disposed`
- 現所有者 id

ライフサイクル:

```text
新車生産 -> 利用中 -> 中古売り出し -> 中古売買 -> 再利用 -> 廃棄
```

## 4. 価格と資金のルール

- 全参加者は初期資金 1000 を持つ
- 新車価格は 180 固定
- 中古車価格は `max(1, 新車価格 - 車齢(月))`
- 中古価格は売り注文時ではなく、買い注文が入った時点の車齢で決まる
- 売買時は買い手の資金を減算し、売り手またはメーカーの資金を加算する

## 5. 月次更新順

MVP の更新順は固定とする。

1. 全個人の年齢を 1 ヶ月加算
2. 3 ヶ月ごとに新しい個人を 1 人追加
3. 全車両の車齢を 1 ヶ月加算
4. 車齢 180 ヶ月以上の車両を廃棄
5. 年齢 840 ヶ月以上の個人を市場から除去し、保有車両を廃棄
6. 未保有かつ 216 ヶ月超の個人に需要発生判定を行う
7. 保有期間 36 ヶ月以上の個人に売却意思判定を行う
8. 売り注文を登録する
9. 買い注文を処理する
10. 中古売買または新車納車を反映する
11. 売却が成立した個人の買い替えを同月に処理する
12. 月次メトリクスを集計する

競合ルール:

- 中古売買のマッチングは FIFO
- 競合時はエージェント ID の昇順で処理
- 寿命到達と売買が同月に重なる場合は廃棄を優先

## 6. イベントログ仕様

イベントは UI 表示用の文字列ではなく、構造化ドメインイベントとして扱う。

イベント種別:

- `AGENT_BORN`
- `AGENT_EXITED`
- `NEED_DETECTED`
- `NEW_CAR_ORDERED`
- `NEW_CAR_DELIVERED`
- `USED_SELL_ORDER_PLACED`
- `USED_BUY_ORDER_PLACED`
- `USED_TRADE_EXECUTED`
- `VEHICLE_DISPOSED_EOL`
- `OWNER_EXIT_DISPOSAL`

イベント推奨フィールド:

- month
- eventType
- agentId
- sellerId
- buyerId
- vehicleId
- price
- memo

## 10. 開発ガイドライン

- シミュレーションロジックは UI と分離する
- ドメインイベントを先に生成し、表示は後段で行う
- ルール変更しやすいように価格決定や売買ロジックを関数単位で分離する
- 再現性のため乱数は必ずシード付きで扱う
- 暗黙の更新順を作らず、月次処理順を固定する
